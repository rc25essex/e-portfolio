Capstone Project

Event-Driven Order Processing System

Project Overview

This capstone project demonstrates the design and implementation of an event-driven order processing system that models how customer orders can be processed asynchronously using loosely coupled services. It was developed to consolidate the advanced object-oriented design, software architecture and secure development principles studied throughout the module while enhancing my professional software engineering skills through the development of a realistic backend application.

The application models two logical microservices—an Order Service and a Fulfilment Service—within a single application for simplicity. Each service has a clearly defined responsibility and communicates through Kafka events rather than direct method calls, demonstrating event-driven architecture and asynchronous messaging without the complexity of deploying multiple applications.

Customers can submit orders through a REST API. Orders are validated, stored with an initial status, processed asynchronously and updated throughout the fulfilment lifecycle. Clients can then retrieve the latest order information through REST endpoints.

Project Objectives

  • Create, validate and store customer orders.
  • Publish order-created events through Kafka.
  • Process orders asynchronously through a fulfilment component.
  • Publish fulfilled or rejected order events.
  • Update and retrieve the current order status.
  • Apply clean code, layered design and automated testing.
  • Support application quality through TDD and BDD testing practices using JUnit 5, Mockito and Karate.

Architecture Style

The project applies event-driven architecture to separate order creation from fulfilment processing. The Order Service publishes events rather than calling the Fulfilment Service directly, reducing coupling between components and allowing each stage of the workflow to operate independently.

Although the two logical services are contained within one Spring Boot application, their responsibilities remain separated through packages, interfaces and event contracts. This keeps the project manageable for local development while reflecting the communication style used by independently deployed microservices.

Event Flow

  1. A client submits an order through the REST API.
  2. The Order Service validates the request and calculates the order total.
  3. The order is stored with an initial PENDING status.
  4. An OrderCreatedEvent is published to Kafka.
  5. The Fulfilment Service consumes and processes the event asynchronously.
  6. The order is either fulfilled or rejected according to the processing result.
  7. A fulfilment result event is published through Kafka.
  8. The Order Service consumes the result and updates the final status.
  9. The client retrieves the latest order state through the REST API.

This workflow ensures that the REST request does not need to wait for fulfilment to complete. The initial response confirms that the order has been accepted, while the final outcome is recorded when the result event is consumed.

Design and Learning

The project was developed incrementally, beginning with functional and non-functional requirements, high-level architecture and UML diagrams. The domain model, REST layer, persistence layer and Kafka components were then implemented in stages, with each layer tested before further functionality was introduced.

This approach made defects easier to isolate and helped maintain clear responsibilities between controllers, services, repositories, event producers and consumers. Unit tests verified individual classes, while integration-focused tests checked REST behaviour, persistence and event handling.

The source code is supported by a High-Level Design document, UML diagrams, test results and example REST responses. Together, these artefacts provide evidence of the complete development process from initial design through implementation, testing and final evaluation.

Key Learning

  • Configuring Kafka producers, consumers and topics within Spring Boot.
  • Understanding asynchronous processing and loose coupling.
  • Designing stable event contracts and order lifecycle states.
  • Integrating REST endpoints, validation and JPA persistence.
  • Using Docker Desktop to run Kafka locally.
  • Applying incremental development and layered automated testing.

The project also strengthened my understanding of the trade-offs involved in event-driven systems, including eventual consistency, message handling and the need for clear event boundaries. It provides a practical foundation for further development in Kafka, microservices, containerisation and distributed system design.
