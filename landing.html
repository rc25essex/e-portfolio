Unit 1–2: Object-Oriented Programming & SOLID Principles

This seminar activity explored the SOLID principles of object-oriented design and how they contribute to developing maintainable, extensible and testable software. The work included a nine-slide PowerPoint presentation, a Python implementation demonstrating each principle in practice, and automated unit tests using pytest to verify the implementation.

Summary

The Python application models geometric shapes such as circles, squares and rectangles using a shared `Shape` abstraction and separate subclasses for each shape. Each subclass is responsible for calculating its own area, while the main program works with the common `Shape` type rather than depending on specific implementations. This design demonstrates abstraction, inheritance and polymorphism while applying the SOLID principles in practice. New shapes can be added by creating another subclass that implements `calculate_area()`, without modifying the existing shape classes. The accompanying presentation explains how each SOLID principle is reflected in the implementation, and pytest unit tests verify that each shape returns the expected area.

Learning & Reflection

Key Learning

  • Applying abstraction, inheritance and polymorphism in object-oriented design.
  • Using the Single Responsibility Principle to separate class responsibilities.
  • Extending behaviour using the Open/Closed Principle.
  • Loose coupling through Dependency Inversion.
  • Verifying behaviour using unit tests developed with pytest.

Developing both the presentation and the accompanying Python implementation strengthened my understanding of how object-oriented design principles can improve software maintainability, extensibility and code readability. Creating unit tests with pytest also reinforced the importance of verifying software behaviour and designing classes that can be tested independently. This experience provided a solid foundation for applying more advanced design patterns and software architecture concepts throughout the remainder of the module.

Unit 3: Creational Design Patterns

Python implementation demonstrating the Abstract Factory pattern with automated unit tests using pytest.

Summary

This artefact explores the Abstract Factory design pattern by implementing a configurable AI service capable of working with different providers without changing the application logic. Separate product families were created for OpenAI and Azure AI, with each factory producing compatible text generation and embedding model implementations.

The client application interacts only with the abstract factory and product interfaces, allowing the underlying AI provider to be changed simply by selecting a different factory. This demonstrates how the Abstract Factory pattern promotes loose coupling, improves maintainability and supports future extensibility. Automated unit tests developed with pytest verify that each factory creates the correct product family and that the application behaves consistently regardless of the selected provider.

Learning & Reflection

Key Learning

  • Applying the Abstract Factory pattern to create related object families.
  • Programming against interfaces rather than concrete implementations.
  • Reducing coupling between application logic and implementation details.
  • Supporting extensibility by introducing new providers without modifying existing client code.
  • Verifying factory behaviour through automated unit testing with pytest.

Developing this application strengthened my understanding of creational design patterns and demonstrated how software can remain flexible when implementation details are abstracted behind common interfaces. Modelling the example around AI providers also illustrated a realistic scenario where different services can be substituted without affecting the rest of the application. Creating automated tests reinforced the importance of validating object creation behaviour and ensuring each factory produces compatible components.

Unit 4: Structural Design Patterns

Python implementations demonstrating several structural design patterns, including Decorator, Composite, Adapter and Bridge, with automated unit tests using pytest.

Summary

This artefact explores how structural design patterns improve software flexibility by organising relationships between objects without modifying existing code. A coffee ordering system demonstrates the Decorator pattern by allowing optional additions such as milk and sugar to be applied dynamically while preserving the original coffee implementation. The Composite pattern is demonstrated through hierarchical menu and file system examples, enabling individual items and groups of items to be managed through a common interface.

Additional implementations include the Adapter pattern, which allows a legacy SOAP-based payment service to be accessed through a modern payment interface, and the Bridge pattern, which separates remote controls from the devices they operate, allowing both hierarchies to evolve independently. Automated unit tests developed with pytest verify the behaviour of the implemented design patterns.

Learning & Reflection

Key Learning

  • Applying the Decorator pattern to extend object behaviour dynamically.
  • Using the Composite pattern to represent hierarchical object structures.
  • Adapting incompatible interfaces using the Adapter pattern.
  • Separating abstraction from implementation using the Bridge pattern.
  • Applying object composition to create flexible and maintainable software.
  • Verifying behaviour through automated unit testing with pytest.

Developing these implementations strengthened my understanding of how structural design patterns improve flexibility, reduce coupling and encourage reusable software components. Implementing multiple patterns also highlighted that different design problems require different architectural solutions, from extending functionality dynamically to integrating legacy systems and separating abstractions from implementations. Creating automated unit tests reinforced the importance of validating each component independently and maintaining confidence when extending or modifying the software.

Unit 5: Behavioural Design Patterns

Python implementations demonstrating the Strategy and Visitor behavioural design patterns, supported by automated unit tests developed with pytest.

Summary

This artefact includes two practical exercises exploring how behavioural design patterns organise communication and responsibilities between objects. The Strategy pattern was applied as part of a discussion exercise that required an existing payment processing system to be analysed and refactored. The original implementation used a growing if-elif chain to process Credit Card, PayPal and Bank Transfer payments, creating tight coupling and requiring the PaymentProcessor class to be modified whenever a new payment method was introduced.

In the refactored version, each payment method is encapsulated within a separate strategy class implementing a common PaymentStrategy abstraction. The PaymentProcessor delegates the payment behaviour to the selected strategy, while a strategy map enables the appropriate implementation to be selected dynamically. This improves readability, reduces coupling and supports the Open/Closed Principle by allowing new payment methods to be introduced without modifying the existing processor.

The Visitor pattern was explored through a standalone employee reporting program created as part of the final assignment exercises. Developer and Manager objects implement a shared employee abstraction and accept a ReportVisitor, which produces a report appropriate to each employee type. This demonstrates how a new operation can be applied across related object types without placing the reporting logic directly inside the employee classes. Automated unit tests using pytest verify the behaviour of both implementations.

Learning & Reflection

Key Learning

  • Identifying maintainability problems caused by long conditional structures.
  • Applying the Strategy pattern to encapsulate interchangeable behaviours.
  • Selecting payment behaviour dynamically at runtime.
  • Reducing coupling between the payment processor and individual payment methods.
  • Applying the Visitor pattern to introduce operations across related object types.
  • Separating reporting behaviour from the employee classes it operates on.
  • Verifying behavioural pattern implementations using pytest.

Completing these exercises strengthened my understanding of how behavioural design patterns assign responsibilities between objects. Analysing the payment system before refactoring helped me recognise that design patterns should be applied to solve a specific design problem rather than added without purpose. Replacing the conditional payment logic with interchangeable strategies created a clearer and more extensible design.

The Visitor exercise demonstrated a different approach by separating an operation from the object structure on which it acts. Although the program is a small standalone example rather than a full employee management system, it clearly illustrates how reporting functionality can be added without embedding it within the Developer and Manager classes. Testing both examples reinforced the importance of verifying each behaviour independently and increased my confidence in selecting an appropriate pattern for different software design problems.

Unit 7: Secure Coding Practices

Collaborative discussion exercise analysing and refactoring an insecure Python authentication system, supported by automated unit tests using pytest.

Summary

This artefact examined security weaknesses in a basic authentication program, including plaintext password storage, weak passwords, missing input validation and unlimited login attempts. The original implementation also relied on direct credential comparison and provided little protection against malicious or unexpected input.

The refactored version hashes passwords using bcrypt, enforces password-strength requirements and validates empty credentials before processing. Secure hash verification replaces plaintext password comparison, while failed login attempts are tracked and accounts are locked after three unsuccessful attempts to reduce the risk of brute-force attacks. Automated tests developed with pytest verify registration, authentication, password validation and account-lockout behaviour.

Learning & Reflection

Key Learning

  • Identifying weaknesses in insecure authentication code.
  • Protecting stored passwords using bcrypt hashing and salting.
  • Enforcing stronger password requirements.
  • Validating credentials before processing user input.
  • Reducing brute-force risk through failed-attempt tracking and account lockout.
  • Testing secure and unsuccessful authentication scenarios using pytest.

Analysing the original implementation strengthened my understanding that functional authentication is not necessarily secure authentication. Refactoring the program demonstrated how password hashing, input validation and login-attempt controls can be combined to provide layered protection. Writing automated tests also reinforced the importance of verifying both successful behaviour and security-related failure conditions.

Unit 8: Refactoring and Code Smells

Collaborative discussion exercise analysing code smells and refactoring a Python pricing function, supported by automated unit tests using pytest.

Summary

This artefact explored how refactoring can improve code readability and maintainability without changing existing behaviour. The original pricing function contained magic numbers for discount values and an if-elif chain that handled several product types within one method.

The function was refactored by replacing the hard-coded discount values with named constants and a dictionary of discount factors. This produced a clearer and more maintainable solution while preserving the original pricing behaviour. The exercise also considered how the Strategy pattern could provide a more scalable alternative if the application later required more complex discount rules. Automated tests developed with pytest verify the calculations for books, electronics, products without discounts and collections containing multiple item types.

Learning & Reflection

Key Learning

  • Identifying magic numbers and conditional logic as code smells.
  • Replacing hard-coded values with clearly named constants.
  • Simplifying conditional logic using a dictionary lookup.
  • Preserving existing behaviour while improving code structure.
  • Selecting a refactoring approach appropriate to the size of the program.
  • Verifying refactored behaviour using pytest.

Completing this exercise strengthened my understanding that refactoring should improve the internal structure of code without changing its external behaviour. Replacing the magic numbers and conditional chain made the pricing logic easier to understand and modify. It also highlighted the importance of choosing a proportionate solution, as the dictionary approach was sufficient for this small program while the Strategy pattern would be more appropriate for a larger system with complex discount rules. Writing automated tests provided confidence that the refactoring preserved the original calculation results.
